## PRIMARY SECURITY CHALLENGE: Age TTY Subversion

### The Core Problem 🚨
**Issue**: `age -p` (passphrase mode) REQUIRES interactive TTY input and cannot accept piped passphrases, making automation impossible for:
- CI/CD systems
- Test suites  
- Automated key operations
- Non-interactive environments

**Impact**: This breaks padlock's automation features and makes testing extremely difficult.

### Research Methods Being Tested

#### Method 1: Script Command Variations
```bash
# Using script(1) to fake TTY environment
script -qec "printf 'pass\npass\n' | age -p -o output.age input.txt" /dev/null
```
**Approach**: Use script(1) to simulate TTY for age
**Status**: Experimental - testing various script flags and redirections

#### Method 2: Expect/Unbuffer Approaches  
```bash
# Using expect to automate interactive input
expect -c "
  spawn age -p -o output.age input.txt
  expect 'Enter passphrase*'
  send '$passphrase\r'
  expect 'Confirm passphrase*'  
  send '$passphrase\r'
  expect eof
"
```
**Approach**: Use expect/unbuffer to handle interactive prompts
**Dependency**: Requires expect package installation

#### Method 3: Socat/PTY Manipulation
**Approach**: Use socat to create pseudo-TTY connections
**Status**: Advanced TTY manipulation research

#### Method 4: Alternative Age Usage  
**Approach**: Find age command variations that avoid TTY requirements
**Status**: Testing different age flags and modes

#### Method 5: /dev/tty Redirection
**Approach**: Direct TTY manipulation through device files
**Status**: Low-level device access testing

#### Method 6: System Call Tracing
**Approach**: Use strace to understand age's TTY detection mechanisms
**Status**: Deep debugging of age behavior

## KEY MANAGEMENT PATTERNS

### Double Wrapped Keys (Production Pattern)
```
14 test keys in double_wrapped/:
- ignition-master-*.key/meta (7 pairs)
- distro-*.key/meta (2 pairs)
```

**Pattern**: Keys are encrypted twice:
1. **First Layer**: Age encryption with master key
2. **Second Layer**: Passphrase encryption for secure storage
3. **Metadata Files**: JSON tracking for each key (.meta files)

### Derived Keys Architecture
**Purpose**: Hierarchical key generation where child keys derive from parents
**Benefit**: Allows key rotation without re-encrypting all data
**Risk**: Parent key compromise affects all derived keys

### Temporal Chain Security
**Purpose**: Time-based key rotation and expiration
**Benefit**: Automatic security refresh and breach containment  
**Implementation**: Likely time-locked key usage

### SSH Delegation Pattern
**Purpose**: Use SSH infrastructure for key management
**Benefit**: Leverages existing SSH key management
**Risk**: SSH-specific dependencies and limitations
