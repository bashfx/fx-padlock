## THE BASHFX BUILD.SH PATTERN

The Build.sh pattern is BashFX's solution for managing "Legendary Scripts" - sophisticated bash programs that exceed 1,000+ lines and approach the complexity limits of shell scripting.

### PROBLEM IT SOLVES
- **Monolithic Complexity**: Single bash files become unmaintainable beyond ~1,000 lines
- **Function Organization**: No natural way to group related functions in bash
- **Collaboration**: Multiple developers can't easily work on same large script
- **Testing**: Difficult to test individual components of large scripts
- **Source Control**: Large diffs and merge conflicts in monolithic files

## PADLOCK'S BUILD ARCHITECTURE

### Parts Directory Structure
```
./parts/
├── build.map              # Assembly instructions
├── 01_header.sh           # Shebang, logo, metadata (19 lines)
├── 02_config.sh           # Configuration & bootstrap (38 lines)
├── 03_stderr.sh           # Logging utilities (74 lines)  
├── 04_helpers.sh          # Core helpers (1,479 lines)
├── 05_printers.sh         # Output formatting (494 lines)
├── 06_api.sh              # Main API functions (4,518 lines) ⚠️
├── 07_core.sh             # Core orchestration (712 lines)
├── 08_main.sh             # Main entry point (68 lines)
├── 09_footer.sh           # Cleanup/final code (7 lines)
└── 99_test.sh             # Test stubs (0 lines)
```

**Total**: 7,409 lines assembled into `padlock.sh`

### Build Map System
```bash
# Format: NN : target_filename.sh
01 : 01_header.sh
02 : 02_config.sh
# ... mapping number to assembly order
```

The `build.map` controls assembly order and allows for conditional inclusion of components.

## BUILD PROCESS

### How ./build.sh Works
1. **Reads build.map**: Determines assembly order and parts to include
2. **Smart Sync**: Updates part files if external versions exist (supports manual editing)
3. **Sequential Assembly**: Concatenates parts in numerical order
4. **Output Generation**: Creates final `padlock.sh` executable
5. **Syntax Validation**: Optional syntax checking of final output

### Padlock-Specific Build Features
- **Auto-versioning**: Extracts version from header metadata
- **Function Ordinality**: Enforces BashFX function hierarchy across parts
- **Conditional Parts**: Can include/exclude parts based on build flags
- **Incremental Updates**: Only rebuilds when parts change

## BENEFITS OF BUILD PATTERN

### Development Benefits
- **Separation of Concerns**: Each part handles specific functionality
- **Parallel Development**: Multiple developers can work on different parts
- **Easier Navigation**: Find functions by logical grouping, not line numbers
- **Modular Testing**: Individual parts can be tested in isolation
- **Version Control**: Cleaner diffs and easier merge resolution

### Maintenance Benefits
- **Reduced Complexity**: Each part stays under ~500 lines (recommended)
- **Clear Boundaries**: Function placement follows ordinality rules
- **Easier Debugging**: Issues isolated to specific parts
- **Documentation**: Parts can be individually documented

### Architectural Benefits
- **Scalability**: Can grow to 3,000-4,000 lines before needing language port
- **Standardization**: Follows BashFX patterns for consistency
- **Reusability**: Parts can be shared between projects
- **Build Automation**: Integration with CI/CD systems

### Function Ordinality Distribution
```
Super-Ordinal:  main(), dispatch(), options()
High-Order:     do_*() functions (API dispatchers)
Mid-Level:      _helper() functions (subroutines)  
Low-Level:      __literal() functions (system calls)
```

## KNOWN LIMITATIONS & TRADE-OFFS

### Limitations
1. **Build Complexity**: Additional build step adds complexity
2. **Debugging**: Stack traces reference final file, not parts
3. **IDE Support**: Limited IntelliSense across part boundaries
4. **Dependencies**: Must maintain build.sh and build.map files
5. **Size Threshold**: Still approaching bash complexity limits (3,000-4,000 lines)

### Part Size Guidelines
- **Ideal**: Under 300 lines per part
- **Acceptable**: 300-500 lines 
- **Concerning**: 500-1,000 lines (06_api.sh = 4,518 ⚠️)
- **Critical**: 1,000+ lines (needs subdivision)

## PORTING THRESHOLD

### RSB Migration Indicator
When BashFX scripts approach 3,000-4,000 lines, they become candidates for porting to **RSB (Rust String Biased)** architecture:
- Padlock at 7,409 lines is **WELL OVER** the threshold
- Should consider Rust port for "Professional Edition"
- Current version could remain as "Community Edition"
