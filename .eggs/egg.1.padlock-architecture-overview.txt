## WHAT IS PADLOCK? 

Padlock is a sophisticated Git Repository Security Orchestrator that transforms any git repository into a secure vault using age-based encryption. Think of it as "enterprise-grade security for your code secrets" - it handles API keys, credentials, sensitive docs, and configuration files that would otherwise be exposed in git history.

VERSION: 1.6.1 (production-ready, actively developed)
STATUS: 98% complete core functionality, some advanced features in development

## CORE SECURITY MODEL

### The "Locker Pattern" 🔐
- **Plaintext locally**: Developers work with normal files in `locker/` directory
- **Auto-encryption**: Git hooks automatically encrypt `locker/` → `.chest/locker.age`
- **Complete opacity**: Only encrypted blobs are committed to git
- **Team sharing**: Simple age public key distribution (no GPG complexity)

### Key Hierarchy (Authority Chain)
```
X (Skull) → M (Master) → R (Repo) → I (Ignition) → D (Distro)
```

**Key Types:**
- **X (Skull Key)**: Ultimate backup authority (passphrase-wrapped)
- **M (Master Key)**: Global master key for all repositories
- **R (Repo Key)**: Individual repository encryption key
- **I (Ignition Master)**: Authority key bridging R to D keys (passphrase-wrapped)
- **D (Distro Key)**: Third-party/AI access keys (passphrase-wrapped)

**"Ignition Keys"** = Any passphrase-wrapped key for secure automation access

## ARCHITECTURE PATTERNS

### BashFX Build System
- **Parts-based**: 9 part files (7,400+ lines total) build into final `padlock.sh`
- **Modular**: Each part handles specific concerns (config, helpers, API, core)
- **Build Map**: Controlled assembly via `parts/build.map`
- **Function Ordinality**: Strict hierarchy (High-Order → Mid → Low-Level)

### Command Structure (CLI Patterns)
```bash
# Mini-dispatchers (entity-specific operations)
padlock master generate
padlock ignite create [name] --phrase="secret"

# Predicate commands (cross-entity operations)  
padlock rotate master
padlock key is skull --path=/path

# Direct operations
padlock clamp /repo
padlock lock / unlock
```

### Security Framework
- **XDG+ Compliance**: Uses `~/.local/etc/padlock/` for configuration
- **State Management**: RC files for session tracking
- **Backup Strategy**: Multiple recovery mechanisms
- **Access Control**: Environment-based automation (`PADLOCK_IGNITION_PASS`)

## TYPICAL WORKFLOW

```bash
# 1. One-time setup
padlock setup                    # Creates master key + ignition backup

# 2. Secure a repository  
padlock clamp /my/repo          # Deploys security to repo

# 3. Normal development
echo "API_KEY=secret" > locker/config/.env
git add . && git commit         # Auto-encrypts via hooks

# 4. Team member access
git clone repo && padlock unlock  # Decrypts for local work

# 5. Automation access (CI/CD)
PADLOCK_IGNITION_PASS="phrase" padlock unlock  # Scriptable access
```

## SECURITY STRENGTHS
- **Multi-layered**: Hierarchical key authority with proper delegation
- **Recovery Options**: Multiple backup mechanisms prevent permanent lockout
- **Automation-Friendly**: Secure environment-based access for CI/CD

## TECHNICAL DEBT & RISKS

### Safety Concerns ⚠️
1. **Permanent Lockout**: If master keys are lost AND ignition backups fail
2. **Test Contamination**: Tests can interfere with live keys
3. **Protocol Violations**: Some operations bypass git safety mechanisms
4. **Development Mode**: Not all encryption uses production-grade methods

### Code Quality Issues
1. **Size Complexity**: 7,400+ lines approaching maintainability limits
2. **Build Dependencies**: Complex part system increases brittleness  
3. **Function Count**: 50+ functions, some overlap in responsibilities
4. **Test Infrastructure**: Needs better isolation and cleanup

## RECOMMENDATIONS

### For Production Use
- **CRITICAL**: Always backup repositories before running padlock operations
- **REQUIRED**: Test in isolated environments first
- **ADVISED**: Use ignition keys for automation, never expose master keys

### For Development
